# E2E Tests

E2E tests spin up real infrastructure using `testcontainers` and exercise the full application stack from HTTP request to database. They require Docker and are meaningfully slower than the other three tiers. Use them for critical paths and smoke tests, not for every feature.

## Prerequisites

Docker must be running. Verify with:

```bash
docker info
```

If Docker is unavailable the tests skip gracefully rather than failing hard -- see [Skipping without Docker](#skipping-without-docker) below.

## Running E2E tests

```bash
pnpm run test:e2e
```

## Basic structure

```typescript
// tests/e2e/app.test.ts
import { describe, it, expect, beforeAll, afterAll } from 'vitest';
import { PostgreSqlContainer } from '@testcontainers/postgresql';
import type { StartedPostgreSqlContainer } from '@testcontainers/postgresql';
import { createApp, loadConfig } from 'telaio';
import type { TelaioApp } from 'telaio/types';

const skipE2e = !process.env.CI && !(await dockerAvailable());

describe.skipIf(skipE2e)('E2E: Application', () => {
  let container: StartedPostgreSqlContainer;
  let app: TelaioApp;

  beforeAll(async () => {
    container = await new PostgreSqlContainer().start();

    const config = loadConfig({
      source: {
        DATABASE_URL: container.getConnectionUri(),
        NODE_ENV: 'test',
      },
      skipEnvLoad: true,
    });

    app = await createApp({ config })
      .withDatabase()
      .withPlugins({ autoload: false })
      .withSchemas(false)
      .build();

    await app.start();
  }, 60_000); // allow up to 60s for container startup

  afterAll(async () => {
    await app.stop();
    await container.stop();
  });

  it('serves the health check endpoint', async () => {
    const res = await app.fastify.inject({ method: 'GET', url: '/healthz' });
    expect(res.statusCode).toBe(200);
  });
});
```

## Skipping without Docker

Use `describe.skipIf()` with a Docker availability check. This lets CI fail loudly if Docker is expected, while local runs without Docker skip gracefully instead of producing confusing errors.

```typescript
async function dockerAvailable(): Promise<boolean> {
  try {
    const { execa } = await import('execa');
    await execa('docker', ['info'], { stdio: 'ignore' });
    return true;
  } catch {
    return false;
  }
}

const skipE2e = !process.env.CI && !(await dockerAvailable());
```

On a CI runner where `CI=true`, `skipE2e` is always `false` so the tests run (and fail loudly if Docker is missing). On a local machine without Docker they are skipped cleanly.

## Running migrations in E2E tests

If your routes depend on database tables, run migrations before starting the app:

```typescript
import { migrateToLatest, runFrameworkMigrations } from 'telaio/db/migrations';
import { createPool, createDatabase } from 'telaio/db';

beforeAll(async () => {
  container = await new PostgreSqlContainer().start();

  const config = loadConfig({
    source: { DATABASE_URL: container.getConnectionUri(), NODE_ENV: 'test' },
    skipEnvLoad: true,
  });

  const pool = await createPool(config);
  const db = await createDatabase(pool);

  // Run Telaio's internal migrations first, then your app migrations
  await runFrameworkMigrations(db);
  await migrateToLatest(db);

  app = await createApp({ config })
    .withDatabase({ pool, db })
    .withPlugins({ autoload: false })
    .build();

  await app.start();
}, 60_000);
```

## Testing with Redis

Add a Redis container alongside Postgres:

```typescript
import { PostgreSqlContainer } from '@testcontainers/postgresql';
import { RedisContainer } from '@testcontainers/redis';
import type { StartedPostgreSqlContainer } from '@testcontainers/postgresql';
import type { StartedRedisContainer } from '@testcontainers/redis';

let pgContainer: StartedPostgreSqlContainer;
let redisContainer: StartedRedisContainer;

beforeAll(async () => {
  [pgContainer, redisContainer] = await Promise.all([
    new PostgreSqlContainer().start(),
    new RedisContainer().start(),
  ]);

  const config = loadConfig({
    source: {
      DATABASE_URL: pgContainer.getConnectionUri(),
      REDIS_ENABLED: 'true',
      REDIS_URL: redisContainer.getConnectionUrl(),
      NODE_ENV: 'test',
    },
    skipEnvLoad: true,
  });

  app = await createApp({ config })
    .withDatabase()
    .withCache()
    .withPlugins({ autoload: false })
    .build();

  await app.start();
}, 90_000);
```

## Full request lifecycle test

```typescript
it('creates a user and fetches them back', async () => {
  // Create
  const createRes = await app.fastify.inject({
    method: 'POST',
    url: '/users',
    headers: { 'content-type': 'application/json' },
    body: JSON.stringify({ name: 'Alice', email: 'alice@example.com' }),
  });
  expect(createRes.statusCode).toBe(201);

  const { id } = createRes.json<{ id: string }>();
  expect(id).toMatch(/^[0-9a-f-]{36}$/); // UUID

  // Fetch
  const getRes = await app.fastify.inject({
    method: 'GET',
    url: `/users/${id}`,
  });
  expect(getRes.statusCode).toBe(200);
  expect(getRes.json()).toMatchObject({ id, name: 'Alice', email: 'alice@example.com' });
});
```

## Timeout guidelines

Container startup time varies. Set explicit timeouts on `beforeAll` hooks to avoid the default 5-second Vitest timeout:

| Scenario | Suggested timeout |
|----------|------------------|
| Postgres only | 30s -- 60s |
| Postgres + Redis | 60s -- 90s |
| Postgres + Redis + S3 (LocalStack) | 90s -- 120s |

Pass the timeout as the second argument to `beforeAll`:

```typescript
beforeAll(async () => { ... }, 90_000);
```

## Cleanup

Always stop containers in `afterAll`. If a test crashes before `afterAll` runs, `testcontainers` will attempt an automatic cleanup when the Node process exits (Ryuk container), but explicit teardown is more reliable.

```typescript
afterAll(async () => {
  await app.stop();
  await pgContainer.stop();
  await redisContainer?.stop();
});
```
