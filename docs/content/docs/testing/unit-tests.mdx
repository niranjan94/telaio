# Unit Tests

Unit tests are colocated with their source files inside `src/**/__tests__/`. They have no infrastructure dependencies -- no database, no Redis, no server. Everything external is mocked with `vi.mock()`.

## File layout

Place unit tests in a `__tests__/` directory next to the module they test:

```
src/
  db/
    client.ts
    __tests__/
      client.test.ts
  cache/
    index.ts
    __tests__/
      index.test.ts
```

Files must be named `*.test.ts`.

## Running unit tests

```bash
pnpm run test:unit
```

This runs Vitest in unit mode without the linter. The `pnpm run test` command runs Biome lint first and then the unit suite -- prefer `test:unit` during rapid iteration.

## Writing a unit test

A basic test for a pure utility function needs no mocking at all:

```typescript
// src/db/__tests__/filters.test.ts
import { describe, it, expect } from 'vitest';
import { applyFilter } from '../query-builders/index.js';
import { Kysely } from 'kysely';

describe('applyFilter', () => {
  it('builds an equality condition', () => {
    // Use Kysely's compile() to test query shape without a real database
    const db = new Kysely<{ users: { status: string } }>({
      dialect: {
        createAdapter: () => ({ ...} as any),
        createDriver: () => ({ ...} as any),
        createIntrospector: () => ({ ...} as any),
        createQueryCompiler: () => ({ ...} as any),
      },
    });

    const query = applyFilter(
      db.selectFrom('users').selectAll(),
      { status: 'active' },
      ['status'],
    );

    const { sql, parameters } = query.compile();
    expect(sql).toContain('"status"');
    expect(parameters).toContain('active');
  });

  it('ignores columns not in the allowlist', () => {
    // filterableColumns does not include 'role' -- should be silently dropped
    const query = applyFilter(
      db.selectFrom('users').selectAll(),
      { role: 'admin' },
      ['status'],
    );

    const { sql } = query.compile();
    expect(sql).not.toContain('"role"');
  });
});
```

## Mocking peer dependencies

Heavy optional dependencies (Kysely, Redis, pg-boss, AWS SDK) are loaded via dynamic `import()`. Mock them with `vi.mock()` at the top of the test file:

```typescript
// src/cache/__tests__/cache.test.ts
import { describe, it, expect, vi, beforeEach } from 'vitest';

vi.mock('ioredis', () => ({
  default: vi.fn().mockImplementation(() => ({
    get: vi.fn().mockResolvedValue(null),
    set: vi.fn().mockResolvedValue('OK'),
    del: vi.fn().mockResolvedValue(1),
    quit: vi.fn().mockResolvedValue('OK'),
  })),
}));

import { createCache } from '../index.js';

describe('Cache', () => {
  it('returns null for a missing key', async () => {
    const cache = createCache({ enabled: true, url: 'redis://localhost:6379' });
    const value = await cache.get('nonexistent');
    expect(value).toBeNull();
  });

  it('disabled mode always returns null without connecting', async () => {
    const cache = createCache({ enabled: false });
    expect(cache.redis).toBeNull();
    const value = await cache.get('any-key');
    expect(value).toBeNull();
  });
});
```

## Mocking the config

Use `loadConfig` with `source` and `skipEnvLoad: true` to avoid touching `.env` files or real environment variables:

```typescript
import { loadConfig } from 'telaio';

const config = loadConfig({
  source: {
    DATABASE_URL: 'postgresql://localhost/test',
    NODE_ENV: 'test',
  },
  skipEnvLoad: true,
});
```

This is safe to call at the module level. The resulting config object is a plain record -- pass it wherever a config is expected.

## Guidelines

- Keep each test focused on one behaviour. A failing test should tell you exactly which line of source code is broken.
- Avoid `beforeAll` for shared state across unrelated tests. Use `beforeEach` or local setup inside `it` blocks.
- Do not start a Fastify server in unit tests. Use `app.fastify.inject()` in integration tests instead.
- If a test requires a real database, it belongs in the integration or E2E tier, not here.
